### 9. CSS 色块冰箱模型不符合真实冰箱观感

现象：

- 用户反馈第一版冰箱模型“根本不像冰箱”，更像色块组合。

原因：

- 纯 WXSS 绘制的分区虽然可点击，但缺少真实冰箱的门、抽屉、隔板、透视和材质细节。
- 该方案无法满足用户对“真实冰箱存放东西感觉”的要求。

处理：

- 放弃 CSS 色块式冰箱模型。
- 改为使用用户提供的真实冰箱模板图片。
- 在图片上覆盖透明分区热区，实现分区查看和添加。

### 10. 图片热区内直接放“查看 / 添加”小按钮过于拥挤

现象：

- 用户反馈图片显示没问题，但每个区域的查看和添加按钮“基本属于乱码”，太丑。

原因：

- 热区尺寸受图片分区限制，直接放两个小按钮会导致文字过小、拥挤、遮挡图片。
- 部分分区本身较窄，小按钮不适合手机触控。

处理：

- 移除图片热区内的“查看 / 添加”小按钮。
- 图片热区只负责点击选中分区，统计信息移到独立标记层。
- 点击任意分区后打开底部清单面板。
- 底部面板内提供大按钮“添加食品到当前分区”和食品编辑入口。

### 11. 冰箱模板图自带标题和“果蔬抽屉”文字影响观感

- 早期整块模糊抽屉会产生暗影，不够自然。
- 最终改为覆盖绿色抽屉面板内部并保留边缘结构，效果更稳定。

### 12. 分区数量徽标在角落且样式突兀

现象：

- 用户反馈每个分区的数量统计标记太丑，位置也不对，不能放在角落。
- 后续确认每个分区统计应展示三项：总数、临期数、已过期数。
- 已过期数即使为 0，也应该始终用红色表达风险项。

原因：

- 旧实现把 `0项` / `1项` 徽标放在每个热区右下角。
- 临期 / 过期提示又是另一个标签，容易和数量徽标叠在一起。
- 部分热区很大或跨左右门，角落位置看起来不像对应真实分区。
- 只展示一个数字会让用户不清楚“总数是否包含临期和过期”。

处理：

- 新增独立的 `zone-stat-marker` 统计点层。
- 数量统计改为 `总 / 临 / 过` 三项。
- `总` 包含临期和已过期。
- `临` 使用首页现有 3 天内到期标准：距离过期日期 0 到 3 天，不包含已过期。
- `过` 不论是否为 0 都固定红色显示。
- 在 `utils/constants.js` 为每个分区配置 `markerPoint`，用百分比手动控制统计点位置。

### 14. 首页全局统计红色卡片曾被挤出可视区

现象：

- 用户反馈三张统计卡片没有真正放在一个页面里，红色“已过期”卡片看不到。

原因：

- 早期用 `button` 实现三张统计卡片，小程序按钮有默认样式和布局影响。
- 在固定一屏方案下，默认按钮尺寸容易把第三张卡片挤出可视区域。

处理：

- 将三张全局统计卡片从 `button` 改为普通可点击 `view`。
- 使用三等分 grid：`minmax(0, 1fr)`。
- 强制卡片 `width: 100%`、`min-width: 0`、`margin: 0`，避免默认按钮样式干扰。

### 15. 分区统计位置方案反复调整

- 用户明确反馈不行，分区单独统计必须加到图片里对应分区。

处理：

- 删除图片下方分区统计网格。
- 恢复冰箱图片内 `zone-stat-marker` 分区统计点。
- 每个统计点展示 `总 / 临 / 过`，并通过 `markerPoint` 定位到对应分区。
- 点击图片内统计点和点击透明热区都会打开对应分区清单。

### 16. 搜索“确定”按钮只改 WXSS 后没有真实缩短

现象：

处理：

- 将搜索“确定”从原生 `button` 改为可点击 `view`。
- 保留原 `bindtap="handleSearchConfirm"`，功能不变。
- 自动化验证修改后按钮真实宽度为 `39px`，输入框宽度恢复到 `320px`。

### 18. 添加页内嵌日历和按钮尺寸导致真机观感混乱

现象：

- 用户真机截图反馈添加页“品类”区域横向溢出，日期区域也显得混乱。
- 用户明确指出日期应该是点击后弹出窗口选择，而不是直接铺在页面中。
- 后续用户要求弹出日历不需要关闭按钮，年月信息不要跨行，上月 / 下月按钮需要缩小。

原因：

- 第一版把日历直接内嵌在添加页里，导致页面高度过长，视觉层级混乱。
- 品类按钮使用三列布局但没有足够限制小程序原生 `button` 宽度，真机渲染时可能撑出卡片。
- 日历翻月按钮第一轮缩小后，自动化读取到真实宽度仍偏大，说明仅设置 `width` 不够，需要同时限制 `max-width`、`min-width` 和 `flex-basis`。

处理：

- 日期区域改成点击日期行后弹出底部日历窗口。
- 生产日期和过期日期共用同一个弹窗日历。
- 点选日期后弹窗自动关闭；弹窗不显示关闭按钮。
- 保留点击灰色遮罩关闭能力。
- 品类按钮改成固定三列，并限制每个按钮宽度。
- 缩小“上月 / 下月”按钮，并加 `flex: 0 0 76rpx` 等限制。
- 年月标题设置单行显示。

失败 / 重试：

- 第一版内嵌日历被真机截图确认体验不佳，已废弃。
- 第一轮自动化验收失败在“上月 / 下月按钮已缩小”，随后继续缩小按钮并限制 flex 后通过。
- 首次运行自动化脚本时，沙盒内启动本地监听端口失败：`listen EPERM: operation not permitted 0.0.0.0`，改用授权后的开发者工具自动化运行。
- 自动化复验时端口 `9451` 被占用，临时改用 `9452` 复验。

验证：

- `node --check pages/item-form/item-form.js` 通过
- `npm run lint` 通过
- `npm run build` 通过
- 微信开发者工具自动化通过 16 项添加页断言：
  - 品类按钮固定在三列卡片内
  - 单位 / 存放位置入口不再显示
  - 日期弹窗初始不显示
  - 点击生产日期和过期日期均能弹出日历
  - 弹窗没有关闭按钮
  - 年月标题保持单行
  - 上月 / 下月按钮已缩小
  - 选择日期后弹窗自动关闭并写入表单数据

后续注意：

- 仍建议真机手动复查日期弹窗高度和日期格子触控面积。
- 如果用户觉得逐月切换日期太慢，再补一个轻量年月选择入口。

## 2026-05-28 “我来选食材”料理碗与缓存记录

现象：

- 用户反馈第一版“小锅”视觉太丑。
- 后续用户指出应先给图片或链接看效果，确认后再改代码。
- 用户进一步询问菜谱页生成内容切走后是否会消失。

原因：

- 第一版小锅使用 CSS 直接画锅盖、把手和蒸汽，质感较粗糙，和“奶油玩具冰箱”风格不统一。
- 当时曾先直接改 WXML / WXSS，没有严格遵守“先展示给我看，再做”的反馈方式。
- 菜谱页 `onShow` 会重新读取库存并生成推荐，原本没有保存“我来选食材”的已选项、临时食材和生成结果。

处理：

- 已撤回未确认的料理碗代码改动。
- 使用 image generation 先生成预览图给用户确认。
- 用户确认后，再把旧小锅替换为奶油陶瓷料理碗视觉。
- 新增当天本地缓存：
  - 生成后写入 `wx.setStorageSync`
  - 切页回来恢复最近一次“我来选食材”结果
  - 点“清空”、切到“菜谱盲盒”或第二天清除缓存
- 临时食材仍不自动写数据库，只在用户点击加入库存并补齐添加页信息后保存。

后续注意：

- 之后遇到用户明确说“先展示给我看，再做”的视觉需求，必须先给图片 / 截图 / 链接确认，再改代码。
- 当前缓存是本地缓存，不能跨设备同步；上线后如需跨设备保留，需要新增云端菜谱历史。

## 2026-05-29 开饭雷达云端 AI 验证记录

### 1. 日历页旧链路会本地生成模板菜谱

现象：

- 检查当前代码后确认，日历页旧实现会调用 `recipeService.getExpiryUsageRecommendations(items)`，本地根据固定菜谱库和兜底模板生成 3 道菜。

- 旧逻辑把“临期去化”体验先做成本地规则推荐，方便第一阶段快速可用。
- 但当前云函数 AI 路径已经存在，继续在日历页展示本地模板菜会误导用户。

- 云端 AI 失败或无未过期库存时返回空菜谱和原因，不回退本地模板。

后续注意：

- 不要恢复日历页本地菜谱兜底。
- 如果云端等待时间过长，优先优化提示和刷新机制，而不是恢复模板菜。

### 2. AI 食材名带状态和数量导致库存匹配失败

现象：

- 第一次云端 smoke test 返回真实 AI 菜谱，但 `availableItems` 为空。
- AI 返回的 `ingredients` 包含 `临期番茄1份`、`鸡蛋1份`、`米饭1份` 等描述。
- 云函数匹配逻辑只做简单字符串包含，导致库存已有的 `番茄`、`鸡蛋`、`米饭` 被误判为缺少。

处理：

- 在云函数和前端 `recipeService` 中增加食材名归一：
  - 去掉 `临期`、`库存`、`已有` 等状态词。
  - 去掉数量、单位和常见修饰。
  - 再与库存食品名和品类匹配。
- 重新部署并复测后，`availableItems` 可正确匹配库存食材。

后续注意：

- AI 输出字段不能完全信任，所有结构化结果仍需要前端 / 云函数二次归一和校验。

### 3. 基础调味品污染缺少食材列表

后续注意：
