# DECISIONS.md

最后更新：2026-05-28

## 本轮技术与产品决策

### 1. 项目主线从 H5 切换为微信小程序

决策：

- 当前主线改为微信小程序“冰箱管家”。
- 旧 React/Vite H5 不再作为当前开发主线。

原因：

- 用户明确要求做微信小程序，不走 H5、Vercel、Supabase、React H5、Next.js 路线。

### 2. 直接使用现有目录

决策：

- 在现有 `/Users/qzt/Desktop/projects/fridge-app` 目录内创建小程序结构。
- 旧文件能不动就不动，暂时不批量删除。

原因：

- 用户明确希望直接用现有目录。
- 项目安全规则禁止脚本批量删除文件或目录。
- 当前通过 `project.config.json` 忽略旧目录，已经不影响小程序运行。

### 3. 第一阶段使用微信小程序原生 JS

决策：

- 页面逻辑使用原生 JavaScript。
- 视图使用 WXML。
- 样式使用 WXSS。
- 不引入小程序框架或编译链。

原因：

- 第一阶段目标是最小可运行 MVP。
- 原生小程序结构最容易在微信开发者工具中直接导入和调试。

### 4. 使用微信云开发 / CloudBase

决策：

- 数据保存到 CloudBase 云数据库。
- 图片、解析、推荐、提醒等后端能力通过云函数预留。
- 不搭独立后端服务器。

原因：

- 用户 PRD 明确要求使用腾讯云 CloudBase / 微信云开发。
- 小程序云开发会自动绑定 `_openid`，适合第一阶段的数据隔离需求。

### 5. 第一阶段所有智能能力先 mock

决策：

- `parseFoodImage` 第一阶段返回 mock 图片解析结果。
- `parseBarcode` 第一阶段返回 mock 条形码解析结果。
- `generateRecipes` 第一阶段使用规则推荐。
- 页面保存前必须进入确认页。

原因：

- PRD 明确第一版不接真实 OCR、真实商品库、真实 AI。
- 保留接口形态，后续第二阶段可以替换真实服务。

### 6. 页面通过 service 层访问业务逻辑

决策：

- 数据操作放在 `services/itemService.js`
- 解析逻辑放在 `services/parseService.js`
- 日历事件和提醒预留放在 `services/reminderService.js`
- 菜谱规则放在 `services/recipeService.js`

原因：

- 避免页面直接堆大量数据库和业务逻辑。
- 后续从 mock 切换真实服务时，优先改 service 层，不大改页面。

### 7. 底部 Tab 主入口从 5 个调整为 3 个

决策：

- 底部 Tab 当前保留：
  - 冰箱
  - 日历
  - 菜谱
- “添加”不再作为底部 Tab 主入口。
- “我的”不再作为底部 Tab 主入口。
- 添加食品收口到点击冰箱图片分区后的分区清单。

原因：

- 用户明确要求最下面添加功能去掉。
- 用户后续确认取消“我的”部分。
- 当前产品核心交互是“先点冰箱分区，再把食品添加到对应分区”，这样添加时天然带上存放位置。
- 现阶段核心入口是首页库存、日历去化和 AI 菜谱，三 Tab 比四 Tab 更聚焦。

### 8. 暂时保留旧 React/Vite 依赖

决策：

- 暂不删除 `src/`、`dist/`、`public/`、`node_modules/`、`package.json` 等旧 H5 相关文件。

原因：

- 用户说“前面做的能用就用，用不了就删掉”，但当前没有必要冒险清理。
- 删除文件应小步确认，不能脚本批量删除。
- 小程序配置已忽略旧目录，保留不会阻塞运行。

### 9. 使用开发者工具自动化端口做验收

决策：

- 开启开发者工具服务端口和自动化端口。
- 临时使用 `miniprogram-automator` 验证小程序流程。

原因：

- 桌面点击模拟器输入框焦点不稳定。
- 官方自动化接口能更可靠地验证页面数据、云数据库写入和页面跳转。

### 10. 测试数据验收后清理

决策：

- 验收期间创建测试食品。
- 验收完成后通过页面删除逻辑清理。

原因：

- 既要验证真实 CloudBase 写入，又不能污染用户正式数据。

### 11. 备案暂停期间继续做不依赖发布链路的功能

决策：

- 微信小程序备案暂时无法提交时，不停在发布链路上等待。
- 继续推进不依赖备案的本地可运行功能和体验优化。

原因：

- 首页空状态、删除提示、表单校验、选择体验、mock 提示等功能都可以在开发者工具和真机预览里继续验证。
- 这些工作不依赖线上发布，也不会引入新的复杂系统。

### 12. 本轮体验优化只改页面层

决策：

- 首页空状态、删除成功提示、添加 / 编辑表单校验只改 `pages/index/*` 和 `pages/item-form/*`。
- 不修改 CloudBase 数据结构。
- 不修改 service 层。
- 不新增项目依赖。

原因：

- 本轮需求属于交互和提示优化，页面层足够完成。
- 维持当前数据结构可以减少风险。
- 小步迭代更容易测试和回退。

### 13. 自动化测试依赖保持临时安装

决策：

- `miniprogram-automator` 继续只临时安装在 `/private/tmp/fridge-automator`。
- 暂不写入项目 `package.json`。

原因：

- 当前自动化测试主要用于本地验收，不是正式测试体系。
- 避免把临时工具变成项目长期依赖。
- 后续如果决定建立正式自动化测试，再单独设计测试目录和 npm 脚本。

### 14. 不为创建数据库索引临时引入新 CLI

决策：

- 本轮不安装新的 CloudBase CLI 来自动创建 `items` 组合索引。
- 2026-05-22 已改为通过 CloudBase 新版控制台确认 `items` 目标组合索引存在。

原因：

- 微信开发者工具 CLI 当前没有暴露数据库索引管理命令。
- 本机没有现成 `tcb` / `cloudbase` CLI。
- 组合索引是一次性配置项，为它临时引入新工具不符合当前小步迭代原则。

结果：

- 集合：`items`
- 索引名称：`openid_expire_created`
- 字段：`_openid` 升序、`expireDate` 升序、`createdAt` 降序
- 该索引已存在，不需要重复创建。

### 15. 添加 / 编辑页使用直接点选替代系统 picker

决策：

- 品类、单位、存放位置在添加 / 编辑页改为直接点选按钮。
- 保持原有字段和 CloudBase 数据结构不变。
- 保留页面层处理，不新增组件库或复杂抽象。

原因：

- 这些选项数量少，直接点选比打开系统 picker 更快。
- 用户在手机上录入食品时可以减少操作步骤。
- 不改数据结构可以降低回归风险。

### 16. 首页冰箱模型使用真实模板图片，不做 3D

决策：

- 首页第一栏冰箱模型改为真实冰箱模板图片。
- 使用用户提供的 7 张冰箱结构图片作为模板资源。
- 不在当前阶段引入 3D、Three.js、Canvas 交互或复杂拖拽摆放。

原因：

- 用户的核心诉求是“看起来像真实冰箱，并能点击分区管理食品”，不是旋转或 3D 展示。
- 2D 模板图片能更快达到真实冰箱观感。
- 3D 在微信小程序里会增加性能、适配和维护成本，不符合当前小步迭代原则。

### 17. 取消冰箱类型选择

决策：

- 当前不做冰箱类型选择。
- 首页固定使用 `DEFAULT_FRIDGE_LAYOUT_KEY` 对应的默认冰箱模板。
- 首页不再读取本地 `selectedFridgeLayoutKey`。
- `pages/profile` 中的冰箱类型选择 UI 和读写逻辑已移除。

原因：

- 用户明确指出类型按键不应该出现在主页面。
- 用户后续确认同时取消冰箱类型选择。
- 对大多数用户来说，主页面应该直接进入库存管理，而不是反复选择冰箱结构。
- 如果后续支持两个及以上冰箱，再重新设计多冰箱数据结构和切换入口。

### 18. 分区操作放到底部清单面板，不直接塞在图片热区里

决策：

- 冰箱图片上的热区只负责选中分区。
- 图片热区不直接放“查看 / 添加”按钮，分区统计使用独立标记层展示。
- 点击分区后打开底部清单面板。
- 添加、查看、按品类清单、编辑食品都在底部面板中完成。

原因：

- 在图片热区里放“查看 / 添加”小按钮会显得拥挤、难看，也不利于触控。
- 底部面板能提供更大的按钮和更清楚的清单结构。
- 这种交互更符合手机端“点区域 -> 展开详情”的使用习惯。

### 19. 取消“果蔬抽屉 / 保鲜”独立区域

决策：

- 不再把“果蔬抽屉”作为独立存放位置或独立热区。
- 新录入食品的存放位置不再出现 `保鲜`。
- 历史 `保鲜` 数据统一归到 `冷藏` 展示和保存。

原因：

- 用户明确指出果蔬抽屉应该和冷藏区合并。
- 当前阶段先保持位置分类简单，减少用户录入时的选择负担。
- 保留历史数据兼容可以避免旧数据在首页分区里消失。

### 20. 分区数量标记独立于热区角落

决策：

- 数量统计不再放在热区右下角。
- 每个分区使用独立浅色统计点展示三项：`总 / 临 / 过`。
- `总` 包含临期和已过期。
- `临` 沿用首页“3 天内过期”标准，即距离过期日期 0 到 3 天，不包含已过期。
- `过` 表示已过期数量，不论是否为 0 都固定红色展示。
- 统计点位置由 `utils/constants.js` 中每个分区的 `markerPoint` 控制。

原因：

- 角落徽标和过期标签容易堆叠，视觉上不自然。
- 真实冰箱图片里的分区不是规则卡片，统计点需要按模板单独摆放。
- 三项统计比单一数量更清楚，用户能直接看到每个分区的风险分布。
- 过期项固定红色可以让用户快速形成稳定识别。
- `markerPoint` 比继续调 CSS 更适合后续按真机截图微调。

### 21. 冰箱模板图片先做无字化处理，不引入 3D

决策：

- 继续使用 2D 冰箱模板图。
- 继续清理图片自带顶部标题、“果蔬抽屉”文字，以及分区内部“冷藏区 / 变温区 / 冷冻区 / 门上储物格”等说明文字。
- 不为这个问题引入 3D、Canvas 或新的复杂渲染方案。

原因：

- 用户当前问题是图片上的文字影响观感，不是交互维度不足。
- 2D 模板图仍然是当前最快、最稳定、最接近真实冰箱观感的方案。
- 复杂渲染会增加小程序性能和适配风险，不符合当前小步迭代原则。

### 22. 首页清单改为弹窗承载，页面固定一屏

决策：

- 首页不再常驻展示底部食品列表。
- 首页底部保留三张全局统计卡片：总食品数、临期、已过期。
- 点击全局统计卡片后，弹出对应食品列表。
- 搜索简化为一个搜索框加一个“确定”按钮，点击后弹出搜索结果列表。
- 首页自身使用固定一屏布局，不做上下或左右页面滚动；食品很多时只允许弹窗内部滚动。

原因：

- 用户要求首页排布放在一整个页面，不要上下左右还能翻。
- 常驻列表会持续拉长首页，不适合当前冰箱图主导的交互。
- 弹窗承载列表能保留编辑、删除、查看能力，同时保证首页主界面稳定。

### 23. 分区统计必须显示在冰箱图片对应分区内

决策：

- 每个分区的 `总 / 临 / 过` 统计必须显示在冰箱图片对应分区里。
- 不再把分区统计放在图片下方独立网格。
- 图片内统计点仍通过 `utils/constants.js` 中各分区的 `markerPoint` 控制位置。
- 点击图片内分区统计点或热区，都打开该分区清单。

原因：

- 用户明确要求“必须加到图片里对应的分区”。
- 这种方式最符合“看冰箱分区管理库存”的产品心智。
- 继续使用 `markerPoint` 可以后续按真机截图逐个微调，不需要改业务逻辑。

### 24. 首页截图反馈按视觉密度优先微调

决策：

- 首页保留“我的冰箱”标题，但去掉标题下方说明文字。
- 冰箱图内容在固定容器内略微上移，减少图片顶部空白。
- 搜索“确定”按钮缩小并用 flex 居中。
- 底部三项全局统计的标题和数字加粗加大。

原因：

- 用户截图反馈蓝色说明文字占空间、冰箱图顶部空档太大、确定按钮过宽且文字观感不居中。
- 当前首页要求固定一屏，视觉密度比说明文案更重要。
- 底部三项统计是核心状态入口，需要比普通控件更显眼。

### 25. 弹窗列表只保留必要信息和操作

决策：

- 分区弹窗、首页三色统计弹窗、搜索结果弹窗的列表项只保留：
  - 食品名称
  - 到期日期
  - 编辑
  - 删除
- 不在列表项里重复显示数量、品类、位置和状态徽标。
- 分区弹窗和首页三色统计弹窗只保留标题，不显示标题下方小字。
- 首页三色统计弹窗顶部不再放关闭按钮，继续通过点击遮罩关闭。

原因：

- 这些弹窗本身已经由分区或状态入口打开，列表项里重复显示位置和状态会显得拥挤。
- 手机端弹窗空间有限，列表清爽比信息堆叠更重要。
- 编辑和删除是当前列表里最直接的操作，应该保持明确可点。

### 26. 搜索确认控件使用可点击 `view`，不再使用原生 `button`

决策：

- 首页搜索栏的“确定”控件改为 `view` + `bindtap`。
- 不再使用小程序原生 `button` 承载这个小尺寸控件。

原因：

- 原生 `button` 有默认最小宽度和内部样式，导致即使 WXSS 写窄，真实渲染仍然偏长。
- 该控件只是提交搜索，不需要原生按钮的 loading、open-type 等能力。
- 使用 `view` 可以稳定控制宽度、居中和视觉密度。

### 27. 冰箱分区统计卡片定位规则区分门架和功能区

决策：

- 所有分区统计卡片都继续显示，并同步放大。
- 冷藏区、冷冻区、变温区等非门架功能区卡片按各自热区中心定位。
- 门上储物格卡片继续使用模板配置里的 `markerPoint`，不参与自动居中。

原因：

- 非门架功能区通常对应明确的大区域，居中更符合“卡片属于这个区域”的视觉判断。
- 门架在图片上通常位于打开的门板上，热区可能跨门板或比较窄，直接居中容易偏离真实储物格位置。
- 保留门架 `markerPoint` 可以按具体模板手动微调，避免统一规则破坏门架观感。

### 28. 添加页只保留必要录入项，日期用弹窗日历

决策：

- 添加 / 编辑食品页不再显示数量、单位和存放位置填写项。
- 创建食品时继续给数量和单位写入兼容默认值：
  - `quantity: 1`
  - `unit: "份"`
- 存放位置由冰箱分区入口带入；编辑旧食品时保留原存放位置。
- 日期录入保留两种方式：
  - 生产日期 + 保质期天数，自动生成过期日期。
  - 直接选择过期日期。
- 日期选择不内嵌在页面中，统一改为点击日期行后弹出底部日历窗口。
- 日历弹窗不显示关闭按钮；点选日期后自动关闭，也允许点击遮罩关闭。
- 品类按钮固定三列，避免原生按钮在真机上横向撑出卡片。

原因：

- 当前添加入口已经收口到冰箱分区，用户不需要再手动选存放位置。
- 第一阶段不强调精确数量和单位，减少字段能降低添加食品的操作负担。
- 用户明确反馈内嵌日历混乱，期望点击窗口弹出的日期选择体验。
- 微信小程序原生 `button` 有默认样式影响，需要对品类和日历翻月按钮做更强的宽度约束。

后续影响：

- 如果后续要支持数量和单位管理，可以放到食品详情或高级编辑中，不建议重新塞回第一屏添加表单。
- 如果用户真机反馈逐月切换日期效率低，再考虑给弹窗增加年月快速切换。

### 29. 智能录入同时支持分区入口和独立入口

决策：

- 首页保留“先点冰箱分区，再添加到该区域”的主交互。
- 分区弹窗内提供手动添加、拍食品、拍包装 / 条码三个入口。
- 首页“我的冰箱”标题右侧新增“智能录入”小按钮，用于不预选分区的拍食品、拍包装 / 条码和拍购物小票。
- 不恢复底部“添加”Tab。

原因：

- 单个食品拿在手上时，先想好放哪个区域再录入，符合用户当前产品心智。
- 购物小票天然是批量录入，不适合强行绑定一个冰箱分区。
- 首页小按钮比底部 Tab 更轻，不破坏当前四个主导航结构。

### 30. 第一阶段打开相机 / 扫码，但解析结果继续 mock

决策：

- 拍食品、拍包装和拍小票会调用相机 / 相册入口。
- 条码会调用小程序扫码入口。
- 本轮不上传图片到 CloudBase 云存储。

- 当前阶段不适合为了一个体验状态新增数据库集合和云端清理规则。
- 本地缓存能先解决同设备、同日恢复体验，风险和改动都较小。

后续影响：

- 上线后本地缓存仍不能跨设备同步，卸载小程序或清缓存后会丢失。
- 如果真实 AI 接入后每次生成有成本，建议升级为云端菜谱历史或草稿集合。

### 73. AI / OCR / 条码接入采用“云函数真实路径 + mock 兜底”

决策：

- `parseFoodImage`、`parseBarcode`、`generateRecipes` 均在云函数中接入真实服务路径。
- 所有真实服务失败时保留 mock 兜底，并返回 `providerStatus`、`fallbackReason`。
- 小程序前端不直接调用 OCR、联网搜索、AI 模型，也不保存任何密钥。

原因：

- 用户已经开通并授权配置腾讯云服务，希望继续补齐预留 AI / 搜索能力。
- 真实服务可能受额度、权限、模型频率、API key 是否启用等因素影响。
- 上线前必须保证外部服务不稳定时，小程序核心流程仍可保存食品、展示菜谱。

后续影响：

- 后续页面可以根据 `providerStatus` 提示“已识别 / 预填待核对”。
- 上线前不要删除 mock 兜底。
- 如果真实 AI 稳定，再考虑降低 mock 出现频率，而不是直接移除兜底。

### 74. 条码和天气搜索暂不默认启用联网搜索

决策：

- 当前保留联网搜索代码路径，但 `FRIDGE_ENABLE_REAL_SEARCH=false`。
- `parseBarcode` 在搜索未启用时继续返回条码 mock 结果。
- `generateRecipes` 在搜索未启用时继续使用本地季节 / 天气 mock 上下文。

原因：

- 当前还没有单独配置联网搜索 API key。
- 条码商品搜索准确性需要单独评估，不能直接当作可靠食品数据库。
- 上线前先保证核心库存和确认页流程稳定。

后续影响：

- 启用真实搜索前，需要配置 API key，并重新 smoke test 条码和菜谱上下文。
- 即使启用搜索，搜索结果也必须经过 AI 结构化和用户确认页，不直接入库。

### 75. CloudBase 工具固定版本安装

决策：

- CloudBase CLI 固定使用 `@cloudbase/cli@3.5.2`，安装到临时目录用于部署。
- CloudBase MCP 固定使用 `@cloudbase/cloudbase-mcp@2.20.1`，安装到 `/Users/qzt/.codex/mcp/cloudbase-mcp`。
- Codex MCP 配置只写本地命令，不写腾讯云密钥。

原因：

- 未固定版本的 `npx ...@latest` 属于不必要的外部代码执行风险。
- MCP / CLI 能提升后续部署、日志查看、环境管理效率。
- 密钥不能固化在本地 MCP 配置里。

后续影响：

- Codex 需要重启或新会话后才能加载新增 MCP。
- 如果后续要升级 CloudBase MCP / CLI，先固定新版本号，再安装。

### 76. 不通过配置追加方式修改云函数密钥环境变量

决策：

- 本轮不使用 CLI 配置更新去改写已有密钥环境变量。
- 对 `parseFoodImage` 和 `parseBarcode` 做代码调整，减少对额外 `FRIDGE_ENABLE_REAL_AI` 开关的依赖。

原因：

- 云函数环境变量里已有密钥，配置更新稍有不慎可能覆盖或泄露。
- 代码侧调整更小、更可控，不需要读取或重写密钥值。

后续影响：

- 后续如需更新密钥，优先在 CloudBase 控制台或受控 MCP 工具中逐项修改。
- 文档只记录变量名，不记录变量值。
