# 冰箱小雷达 · 竞品迁移 + 商业化产品规划（V1 → V3）

最后更新：2026-06-14
视角：站在「冰箱里有啥」重度用户「怎样迁移过来、且再也不想回去」的角度，叠加商业化落地。
背景：合规问题正在解决（主体注册中）。AI、拍照识别、广告、会员费将在企业/个体主体落地后分阶段上线。
本版修订：整合了一份外部商业化方案（下称「商业化方案」）的执行层建议；其中的 2026 微信政策已于 2026-06-14 联网核实（见 §9 来源），并对存疑/冲突处做了批判性标注（见 §14）。

> 两条认知声明：
> 1. 对竞品「冰箱里有啥」的判断基于该品类头部产品通常形态（食材联想快录 + 有啥做啥菜谱 + 到期提醒 + 食材库），如与实际有出入需校正。
> 2. 本文 2026 年微信政策已于 **2026-06-14 联网核实**（来源见 §9）：虚拟支付、深度合成类目、AI 标识办法、AI 成长计划、个体户税务**均获二手信源证实**；仅 **iOS 精确费率口径**与**成长计划免费额度是否覆盖视觉识别模型**两点仍需在微信公众平台后台最终确认。

## 0. 不可动摇的核心战略判断

1. 竞品护城河 = **录入省力** + **有啥做啥菜谱**。正面对位 = 必输。
2. 迁移**只发生在「某个高频时刻明显更省心」**，不靠功能数量。
3. 全产品只有两条值得 all-in 的主线，三版都是其递进：
   - 主线一 **录入省力**：V1 本地词库 → V2 拍照识别 → V3 高频额度/多冰箱
   - 主线二 **决策外包（开饭雷达）**：V1 雷达上首页 → V2 AI 接管 → V3 深度建议
4. AI/拍照是这两条主线的**升级形态**，不是另起炉灶抄竞品菜谱列表。
5. **变现与护城河分工（本版新增的关键判断）**：
   - **拍照识别 = 变现引擎**：录入摩擦最痛，用户最愿为「省力」付费 → 它是会员转化的核心理由。
   - **开饭雷达 = 差异化护城河**：决策外包让人记住你、回不去 → 它是留存与口碑的核心，且纯本地、审核安全，永久免费。
   - 一句话：**靠省力（拍照）赚钱，靠省心（雷达）留人。**
6. 「干净」（无广告骚扰、不强制关注、数据本地感）是迁移后「回不去」的隐性原因。V3 变现必须克制，否则亲手摧毁它。

收口：**录入不倒退，迁移才可能；录入领先 + 决策被外包，才谈得上「回不去」。**

## 1. 目标用户与冷启动（本版新增 · 取自商业化方案）

**首批最易冷启动用户：囤货型家庭采购者。**
画像：25–40 岁、负责家里买菜（多为女性），山姆/Costco/电商大促一次囤一后备箱；核心痛点是「囤了忘吃、过期扔掉心疼钱」。
为何选他们：在小红书聚集于「冰箱收纳 / 囤货清单 / 过期断舍离」话题，内容获客成本低；且天然爱晒图——你的「5 分区货架 + 奶油玩具冰箱」UI 本身就是可传播素材。
第二梯队：独居做饭青年。先不专门服务。

**冷启动场景（一句话）**：周六大采购回家，拍 3 张照全部入库 → 周三收到「鲜牛奶后天到期」微信提醒 → 点进去看「用临期食材今晚能做什么」。

**价值主张：不讲大叙事，给用户算两笔账。**
- 时间账：大采购 15 件，手动录入 ~10 分钟 vs 拍 3 张照 ~30 秒。
- 金钱账：一盒鲜奶 15 元、一盒虾 40 元，会员费个位数；每月少扔一样就回本。
- 付费页文案写「一个月少扔一盒牛奶就回本」，**不要写「解锁高级功能」**。

> 评注：这套 GTM 是本产品此前缺的一环，直接采纳。注意它与 §0 主线一致——囤货党的头号痛点正是「录入省力（拍照）+ 提醒不忘」，验证了拍照作为变现引擎的判断。

## 2. 三版本节奏总览

| 版本 | 主题 | 触发条件 | 一句话目标 | 绝不做（本版） |
|---|---|---|---|---|
| **V1** | 体验打磨 + 留存钩子 | 现状即可，个人主体内 | 敢试 + 第一次就省心 + 用提醒留住 | AI、拍照、广告、会员、登录 |
| **V2** | 能力升级（freemium 就位） | 主体/类目/AI 资质落地 | 拍照零摩擦 + AI 接管雷达，**上线即带配额墙** | 把基础能力做成付费墙 |
| **V3** | 开闸收费 + 谨慎变现 | 虚拟支付商户号通过、留存验证 | 会员变现，**优先会员、缓做广告** | 开屏/插屏广告、诱导关注、回收已免费能力 |

> 与商业化方案的节奏分歧（重要）：该方案把 AI+菜谱+会员压成一个 14 天冲刺同时上。这与你定的「V1 改善→V2 加功能→V3 收费」三段式冲突。**调和方案见 §5.0**：AI 一上线（V2）就是 freemium（免费额度 + 会员不限次），会员**代码与配额墙同期就位**，但真正开闸收费等虚拟支付商户号审核通过（落入 V3）。这样既不违背你的节奏，又避免「V2 全免费送 AI、V3 再收回」的红线（§6）。

## 3. 两条主线（脊柱图）

```
录入省力   V1 本地词库联想 → V2 拍照识别(识别即一键批量入库) → V3 高频额度/多冰箱
决策外包   V1 雷达上首页(一句先吃啥) → V2 AI 接管雷达(一句定案+理由) → V3 深度建议/浪费报表
```
任何不落在这两条线上的功能，默认不做。

## 4. V1 · 体验打磨 + 留存钩子（合规内，现在就做）

目标：竞品用户**敢试**（第一屏不被劝退）+ **第一次就省心** + **被提醒留住**。

### P0（本周，1–3 天见效）
- [ ] 修 `pages/index/index.js` 的 `handleZoneAdd` 带 `smartRecommend=1`（分钟级，先让已写好的分区推荐生效）
- [ ] 本地食材词库（80–150 条）+ 名称联想自动填充（品类/默认保质期/自动算过期/分区/图标），复用 `utils/visualAssets.js`；目标 **2 次点击落库**
- [ ] 开饭雷达搬上首页第一屏：去 `%`、`explanation` 改「先用掉：X、Y、Z」（`priorityItems` 已算好）
- [ ] 空状态改「30 秒装满样板冰箱」：30–50 高频食材网格，多选→批量 `createItem`

### P0+（留存钩子，本版上调为 V1 必做）
- [ ] **订阅消息临期提醒**：实装 `cloudfunctions/sendExpiryReminders` 定时扫描 + `reminders` 防重发；添加/编辑保存成功后 `requestSubscribeMessage`；提醒提前天数（先固定 1/3 天）

> 为何上调：没有提醒，本产品就是「需主动打开才有用」的一次性玩具——这是商业化方案最该采纳的一条修正。订阅消息个人主体可用，`sendExpiryReminders` 骨架已在，值得为它多花 2–3 天再提审。模板申请 **D1 就发起**（审核要时间），实装排在录入/雷达之后但**仍在 V1 内收口**。

### P1（1–2 周）
- [ ] 词库扩到 300–500 + 拼音首字母 + 「最近常加 / 再加一次」
- [ ] 雷达文案随时段/状态变化 + 卡内一键「已处理」

### V1「回不去」体验点
- 「只打一个字，剩下全帮我填好」→ 回竞品手动选品类嫌烦（省力上瘾）。
- 打开就被告知「今天先解决这 3 样」→ 回竞品自己翻菜谱嫌累（决策外包）。
- 「香菜明天就坏了」的提醒救了一次浪费 → 它替我记住了我忘的事（外部记忆）。

### V1 不做（本版不做，非永久）
AI、拍照、扫码、广告、会员、登录；菜谱 tab 维持收敛/让位，不用规则菜谱硬碰竞品库。

### 资源极限只做一件
**词库联想录入（含修 `smartRecommend`）** —— 迁移漏斗第一道闸门，纯本地零合规、1 天 MVP，唯一能把录入从劣势翻成优势的杠杆。

## 5. V2 · 能力升级（主体/资质落地后）

定位：用 AI 与识别把两条主线推到竞品做不到的程度，**上线即 freemium**。

### 5.0 freemium 配额墙（调和节奏的关键）
AI 能力一上线就分层，**配额与会员判断写在云函数侧**（前端拦截 = 形同虚设）：
- 免费额度（建议锚点，以实测调）：拍照识别 ~10 次/月、AI 菜谱 ~1 次/天。
- 会员不限次（后台设公平上限防滥用）。
- 支付未开通期间，超额引导位显示「即将开放」占位，**绝不导流到外部/客服/H5 收款**。
→ 这样 AI（V2）天然带商业化骨架，V3 只是把支付开闸 + 扩展付费矩阵。

### 5.1 拍照识别（变现引擎 · 录入主线终极形态）
- 必须做到：识别后**「已识别 N 样 → 一键全入库 + 自动配雷达建议」**，多食材批量、可勾选/编辑/删除，不逐个确认。
- 与雷达缝死：入库即刷新「今天先吃啥」。
- 兜底：识别失败/弱网回退手动表单，不阻断。
- 回不去：拍一张小票几十样秒入库还排好今日处理顺序——竞品逐条确认再也回不去。

### 5.2 AI 接管开饭雷达（护城河 · 决策外包升级）
- **要做**：结合库存+临期+节气+口味，输出**一句定案 + 理由**：「今晚做麻婆豆腐，正好用掉明天到期的豆腐和那把蔫香菜」。
- **不要做**：平铺 3 张菜谱卡让用户重新挑（那是把竞品的决策负担搬回来）。
- **对商业化方案『3 道菜』的修正**：即便展示 3 道，也要有一个**默认主推 + 理由**置顶，其余作为备选折叠，保住「决策外包」心智。
- 合规：AI 调用在云函数侧，key 只放环境变量；AI 生成处加「内容由 AI 生成，仅供参考」标识。

### 5.3 V2 必须埋好的前瞻
- 分清「免费基线 / 未来增值」，**不要把高价值 AI 先全免费送出再回收**（§6 红线）。
- `recipeCache` 按「库存指纹」缓存 24h：同样的冰箱不重复扣额度、不重复花钱。

### V2 不做
社交/晒冰箱/UGC；多冰箱/分区 DIY 深水区（留作 V3 第二付费理由）；数量/单位精细管理。

## 6. V3 · 开闸收费 + 谨慎变现（推广起量、留存验证后）

定位：**为「省更多力」收费，不为「解锁基础」收费**；守住「干净」护城河。

### 6.1 会员「雷达 Pro」（权益克制，建议 3 条）
1. 拍照识别不限次（含一图多食材批量入库）——手动党转化核心理由。
2. AI 菜谱不限次 + 「临期优先」模式（自动把 3 天内到期食材排进今晚）。
3. 提醒增强：自定义提前天数（1–7 天）+ 每日早晨临期汇总一条。

### 6.2 收费扎在哪 / 不扎在哪
| 必须永久免费（动它=自杀） | 可作为增值收费 |
|---|---|
| 手动 + 词库录入 | AI 拍照不限次 / 高频额度 |
| 到期提醒（基础） | AI 菜谱不限次 + 临期优先 |
| 开饭雷达（本地基础版） | 多冰箱 / 家庭共享（V3 第二理由） |
| 库存管理与查看 | 浪费统计报表 / 历史回顾 |
原则：免费部分必须能独立完成「减少浪费」闭环（录入+提醒+雷达）。

### 6.3 定价（建议锚点，非定死，以测试为准）
- 月卡 ~9.9 元、年卡 ~68 元（折合 ~5.7/月，**主推年卡**）、首月 ~6.9 拉新。
- **不做连续包月**（虚拟支付下实现复杂、投诉率高）、**不做终身卡**（断自己后路）。

### 6.4 广告：优先级下调（本版判断）
- 商业化方案早期主张**靠会员、暂不做广告**——本规划采纳：V3 **以会员为主，广告缓做或仅做原生场景位**（临期食材采买、菜谱食材采购），且**会员免广告**。
- ❌ 永不做：开屏 / 插屏 / 强制激励视频 / 诱导关注。这些会一夜击穿「干净」护城河。

### 6.5 红线（最反直觉、最重要）
**已免费放出的核心能力，V3 不能收回付费墙**——回收比从未提供更伤留存。故 V2 设计时就按 V3 蓝图划好免费边界（见 §5.3）。

## 7. 技术改造与数据模型（V2/V3 落地参考 · 取自商业化方案，已据真实代码校正）

> 架构不变：原生小程序 + 微信云开发。坚持不做 Vercel/Supabase/独立后端（境外域名过不了合法域名校验，自建还要域名+备案，云开发是最低成本路径）。

**前端模块**
- `pkg-add/item-form`：加「拍照识别」入口（`chooseMedia` 压缩≤1080 → 传云存储 → 调函数 → 预填表单，可改可删）。
- 多食材 → 新页 `pkg-add/parse-confirm`（列表勾选/编辑后批量保存）。
- `pages/recipes`：占位 → 真实 AI 菜谱页（默认主推 + 备选，标注用了哪些库存、缺什么）。
- `pages/index`：识别快捷入口 + 会员角标。
- 新建分包 `pkg-member`：会员介绍/支付/提醒设置/服务协议。**主包已 1.70MB、距 2MB 红线仅 ~300KB，新页面/新图一律进分包或云存储。**

**云函数（骨架已在 → 实装）**
- `parseFoodImage`：入参 fileId，出参严格 JSON 数组（name/category/zone/quantity/shelfLifeDays/expireDate），schema 校验、重试 1 次、失败兜底返回空；内嵌配额校验。
- `generateRecipes`：入参库存+临期，出参菜谱 JSON；强约束「只能用提供的清单」+ 后端过滤越界食材；24h 库存指纹缓存。
- `sendExpiryReminders`：定时 09:00 扫描 `expireDate <= today + remindDays` 且未提醒，发模板、落库防重发。
- 新增 `paymentDeliver`（虚拟支付发货回调，幂等写 orders、延长 memberUntil）、`getMemberStatus`。

**云数据库**
- 现有：`items / reminders / parseLogs / fridgeZoneConfigs`。
- 新增：`users{_openid, memberUntil, aiQuotaUsed, quotaMonth, remindDays, createdAt}`、`orders{orderId, openid, sku, amount, status, platform, ts}`、`recipeCache`。
- **校正**：`items` 的 `source` 与 `imageFileId` 字段**现有 schema 已存在**（见 `PROJECT_CONTEXT.md` 的 `FridgeItem`），无需新增，拍照链路只需真正写入。

**安全红线（无论仓库公开与否都必须做）**
- 所有集合改「仅创建者可读写」；所有云函数入口用 `cloud.getWXContext()` 取 openid 做鉴权与配额，**不信任前端传入的身份/配额**——否则免费 AI 额度会被脚本刷穿。
- API key 只放云函数环境变量。
- 引入云端数据后，「清空全部食品」按钮加更重确认。
- `chooseMedia`/相册/摄像头/订阅/支付权限同步进隐私保护指引，否则审核必拒。

> 校正：商业化方案称「复用 NEXT_VERSION_GUIDE 里 7 类做法标签配图」——**该资产不存在**，菜谱图早已删除（见 `DECISIONS.md`/`PROJECT_CONTEXT.md`），`visualAssets.js` 是**食材**品类桶图，非菜谱做法图。V2 菜谱页配图需重新设计，且受主包体积约束（优先少图/云存储/复用品类图）。该方案还误以为仓库公开，实际为私有（`DECISIONS.md` #60）；但安全规则收紧的建议依然正确。

## 8. 成本结构（已核实 2026-06-14，来源见 §9）

| 项 | 量级 | 说明 |
|---|---|---|
| AI 文本（菜谱） | 趋近 0 | AI 成长计划 2026 全年送混元 2.0 文生文 1 亿 token + 文生图 1 万张；工具类小程序符合资格 |
| AI 视觉（拍照识别） | ⚠️ 待确认 | 识别食材需视觉/多模态模型；成长计划免费额度主打文生文/文生图，**是否覆盖视觉识别需向云开发确认**，否则按调用付费（约几十~几百元/月级） |
| 云开发 | 0 ~ 20 元/月 | 成长计划新环境免费 6 个月 / 老环境领抵扣券；基础套餐约 20 元/月级 |
| 微信认证 | 300 元/年 | 主体认证标准费用 |
| 虚拟支付抽成 | 安卓 ~1%（2026 限时，标准 10%）；iOS ~12–15% | iOS 为苹果佣金，腾讯 5% 平台技术服务费 2026 优惠期免收；iOS 精确口径以后台为准 |
| 主体税务 | 个体户早期 ≈ 0 | 小规模季销 30 万内免征增值税（2026.1–2027.12）；个体户经营所得 200 万内个税减半 |

要点：不需要 Supabase/Vercel/域名/备案。最大成本变量是**拍照识别的视觉模型是否在免费额度内**——而拍照正是变现引擎，V2 立项时应第一时间向云开发确认。

## 10. 全程警惕的陷阱
1. 菜谱列表对位竞品：V2 也只做「一句定案+理由（+折叠备选）」，不平铺让用户挑。
2. 拍照识别同质化：必须与雷达缝死（识别→一键入库→今日建议），否则只是追平。
3. 社交/UGC：长期默认不做。
4. 多冰箱/分区 DIY：数据证明刚需前不做；留作 V3 第二付费理由。
5. 数量/单位精细管理：已砍，不要加回。
6. 把「个人开发者上线记录」当迁移主打：那是给同行/媒体的故事，仅作点缀。
7. 过早免费送出未来想收费的能力：见 §6.5。
8. 把基础提醒做成付费墙：留存自杀。
