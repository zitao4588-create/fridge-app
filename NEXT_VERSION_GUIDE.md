# NEXT_VERSION_GUIDE.md

最后更新：2026-06-12

## 用途

本文件用于指导「冰箱小雷达」后续版本更新。

后续每次准备做新版本前，先读：

- `AGENTS.md`
- `PROJECT_CONTEXT.md`
- `TODO.md`
- `DECISIONS.md`
- `BUG_NOTES.md`
- `NEXT_VERSION_GUIDE.md`

再决定本轮最小可交付范围。

## 当前版本基线

当前基线是 v1.0 个人主体稳定版：

- 纯微信小程序原生项目
- 不含 AI / 深度合成
- 不含拍照识别入口
- 不含支付
- 不含登录
- 不含 UGC
- 首页手动管理库存
- 日历页查看到期食品
- 开饭雷达为纯本地算法
- 菜谱页为占位页

下一版本应优先稳定当前线上体验，不要马上恢复大功能。

## v1.1 建议主线

v1.1 优先做「上线后稳定性 + 文档 / 截图 / 隐私一致性」。

建议范围：

1. 真机检查首页、日历、菜谱、添加 / 编辑流程。
2. 修复真机发现的小 UI 问题。
3. 更新 README 截图，让截图和当前 v1.0 UI 一致。
4. 检查微信后台隐私协议、服务类目、服务内容是否和当前功能一致。
5. 检查食材图片归桶是否正确。
6. 评估订阅消息到期提醒，但先只做方案，不直接开启真实推送。
7. 继续控制主包体积。

## 图片资源治理原则

当前已经完成一轮图片治理：

- 食材图按品类 / 食材族群共用写实图
- 不再为每个具体食材维护单独本地图
- 删除旧菜谱图片和自动生成占位图
- `utils/visualAssets.js` 负责归桶
- 当前主包约 1.70MB

后续原则：

- 不要无控制新增 PNG
- 新增图片前先估算主包体积
- 优先复用品类桶图
- 如果必须补图，按小批次补
- 补图后做真机检查

当前食材图优先级：

1. 用户照片：`source === "photo"` 且有 `imageFileId`
2. 食材名称关键词归桶
3. 品类兜底
4. 通用默认图

## 云存储方向

用户真实拍照图未来适合存 CloudBase 云存储，并用 `imageFileId` 绑定单条 `items` 数据。

但 v1.0 当前不开放拍照识别入口。

如果企业主体版本恢复拍照识别，建议小步做：

1. 恢复用户选择照片 / 拍照入口。
2. 上传单张图片到 CloudBase 云存储。
3. 写入 `source: "photo"` 和 `imageFileId`。
4. 首页自动优先显示用户照片。
5. 删除食品时再评估是否同步删除该单张云文件。

注意：

- 不要批量删除云文件。
- 孤儿图片只能先做扫描报告，不能直接清理。
- 云存储清理必须逐项确认。

## 订阅提醒方向

v1.x 可以评估到期提醒：

- 先确认微信订阅消息模板和授权流程
- 先设计用户触发授权入口
- `sendExpiryReminders` 当前仍应保持占位或灰度
- 不要在未确认模板、授权、失败重试前直接开启真实推送

## v2 之后再考虑

以下能力不属于 v1.x 个人主体稳定版：

- 拍照识别
- 包装 OCR
- 小票识别
- AI 菜谱
- 天气 / 气候菜谱
- 支付 / 会员

这些能力涉及 AI / 深度合成、OCR、支付或更复杂审核要求。恢复前应先确认主体资质和微信类目，原则上进入企业主体版本后再做。

## 禁止混入的内容

v1.1 不要混入：

- 恢复 AI 菜谱
- 恢复拍照识别
- 恢复条形码扫描
- 接入支付
- 新增登录页
- 新增 UGC
- 引入独立后端
- 引入 Docker
- 引入 Vercel / Supabase / Next.js
- 大规模重构数据结构
- 批量删除本地文件或云端资源
